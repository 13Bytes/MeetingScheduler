Implementation plan

User Session and Identity Refactor

This plan adapts the app from email-only recovery toward a canonical server-side user model. Every user gets a durable anonymous userId, local membership tokens remain useful, and verified email addresses can later recover the same user and all attached meetings.

Shared Decisions

Canonical User

Add a Convex users table. The browser represents the current user with a signed HttpOnly cookie, likely ms_user_session.

Email Ownership

Each normalized email belongs to exactly one canonical user. One user may own multiple email addresses.

User Merge

If a current anonymous user verifies an email already owned by another user, migrate memberships and emails to the existing canonical user, set the browser session to that user, then delete the old user row.

If both users have a membership for the same meeting, keep the voting set linked to the user being merged in. Move that user's email to the canonical user, and move only the non-colliding memberships.

Private Links

Opening a private return link imports only that specific membership into the current user. It does not import unrelated meetings.

Session Lifetime

User sessions last 90 days and are refreshed on relevant interactions. Reuse MEETING_SCHEDULER_IDENTITY_SESSION_SECRET for signing.

Email Session

ms_email_session becomes derived from the canonical user and its verified emailIdentities. User identity becomes the source of truth.

Target Behavior

  • A new anonymous user gets a server-side userId when the app reaches an identity-relevant interaction.
  • Creating a meeting attaches the creator's admin membership to the current userId.
  • Joining or saving availability attaches that membership to the current userId.
  • Locally remembered membership tokens are still stored per meeting, as they are today, but All Meetings imports them by verifying the tokens server-side.
  • Adding an email to the same session links every membership owned by the current userId to the verified email's canonical user.
  • Later verification of that email from another browser restores the same canonical userId and therefore the same All Meetings list.
  • Local membership import runs on All Meetings and on private return-link opens.

Data Model Changes

Table Change Purpose
users Add table with fields like createdAt, updatedAt, optional lastSeenAt. Canonical anonymous-or-verified app user.
memberships Add optional userId: v.id("users") and index by_user. All Meetings can query memberships connected to the current user.
emailIdentities Add optional userId: v.id("users") and index by_user. Keep by_normalized_email. Email verification resolves to a canonical user; multiple emails can belong to one user.
auditEvents Optionally include user merge and membership import audit events. Existing schema can store these as metadata on meeting-level events where a meeting is involved. Debuggability and future support visibility.

This project will reset the Convex database for this refactor instead of running compatibility migrations. The implementation can update the schema directly to the target shape, then reset development data before testing.

Implementation Order

1. Add User Session Primitives

Create a server-side utility similar to lib/identity-session.ts for a signed ms_user_session cookie.

  • Payload should contain userId, issuedAt, and expiresAt.
  • Use MEETING_SCHEDULER_IDENTITY_SESSION_SECRET as the signing secret.
  • Set the max age to 90 days.
  • Refresh the cookie on relevant interactions that ensure or use the user session.
  • Use the same secure-cookie behavior as the current email session cookie.
  • Provide helpers to create, verify, set, and expire the cookie.

2. Add Convex User Functions

Add functions that let trusted Next.js routes ensure and resolve the current user.

  • ensureUser: create a users row if no valid user exists.
  • readUserSession: verify that a session user still exists.
  • mergeUsers: migrate source emails and non-colliding memberships into target user, preserve source-user voting sets for same-meeting collisions, then delete the source user.

These should be protected with the existing internal secret pattern used by identity routes.

3. Introduce Next.js User Session Route

Add a route like /api/user/session that ensures a user session when identity-relevant UI needs one.

  • If ms_user_session is valid, return it.
  • If missing or stale, create a Convex user and set the cookie.
  • Do not create users for homepage-only or static browsing.
  • Refresh the 90-day user session cookie whenever this route confirms or creates a user.

4. Attach New Meetings to Current User

Update meeting creation so the admin membership receives the current userId.

  • Preferred path: route meeting creation through a Next.js API route that can read or create ms_user_session.
  • The route calls a Convex mutation with an internal secret and the resolved userId.
  • Keep the existing client redirect to the private admin membership link.

5. Import Local Membership Tokens

Add a client component on All Meetings that reads locally remembered membership tokens and sends them to a server route for import.

  • Create /api/user/import-memberships.
  • The browser sends raw membership tokens already stored locally.
  • The server ensures the current user session.
  • Convex resolves each token using the existing hashed-token lookup.
  • Valid, active memberships get patched with userId.
  • Invalid or revoked tokens are ignored or returned as non-fatal import failures.

6. Make Private Return Links Import One Membership

When the user opens /join/[membershipToken], ensure a user session and attach only that resolved membership to the current user.

  • Keep storing the token locally for per-meeting return behavior.
  • Do not import any other local tokens from this page.
  • Preserve existing authorization rules: the token remains the authority for editing that membership.

7. Rework Email Verification Around Users

Update email verification so verified emails resolve to canonical users.

  • Email request creates or finds emailIdentities by normalized email.
  • Magic-link verification ensures a current user session if needed.
  • If the email has no userId, assign it to the current user.
  • If the email already has the same userId, just refresh session state.
  • If the email has a different userId, merge the current user into the email's canonical user using the collision rule above.
  • After verification, set the user session cookie to the canonical user. Any email-session display state should be derived from that user and verified email identities.

8. Update All Meetings Query

Change All Meetings from querying by emailIdentityId to querying by userId.

  • Use memberships.by_user as the primary index.
  • Continue showing role, saved response status, lifecycle state, and public poll link.
  • Keep recovery-link actions where they still make product sense.
  • Show verified emails attached to the user as account context, not as the membership source of truth.

9. Update UI Copy and States

Make the product language match the new model.

  • All Meetings should explain that meetings are tied to this browser session and recoverable after email verification.
  • The Email Recovery box should say that adding email links the current session's meetings.
  • The private membership page should say the return link adds this meeting to the current session.
  • Empty state should mention creating, joining, or opening a private return link.

10. Reset Database and Regenerate Convex Types

This refactor intentionally avoids migration compatibility work. Reset the development Convex database after landing the schema change, then regenerate Convex generated files and test with fresh data.

  • Remove migration/backfill code from scope.
  • Document the reset command used for the active Convex deployment before running it.
  • Recreate test meetings through the app after reset.
  • Confirm old anonymous rows are not expected to survive the refactor.

11. Tests

Add focused tests before broad refactors become hard to reason about.

  • Unit tests for user session cookie signing, verification, expiry, and tamper rejection.
  • Convex tests for user creation, membership import, email assignment, and merge behavior.
  • Route tests for email verification adopting the canonical user.
  • Component tests for All Meetings importing local tokens and rendering imported meetings.
  • Regression tests for private return links importing only one membership.

Resolved Implementation Details

  • ms_email_session becomes derived from users and emailIdentities; ms_user_session is the durable session source of truth.
  • For duplicate same-meeting memberships during merge, keep the voting set linked to the user being merged in. Move the linked email to the canonical user along with all memberships that do not collide.
  • User sessions last 90 days and are refreshed on relevant interactions.
  • Local import runs on All Meetings and on return-link opens.
  • No Convex migration/backfill is needed because the database will be reset for this refactor.

Risks and Guardrails

  • Do not authorize access from userId alone until membership ownership rules are explicit. Existing membership tokens should remain the authority for editing a membership.
  • Do not trust client-submitted meeting IDs during import. Accept membership tokens, resolve them server-side, then attach resolved memberships.
  • Keep import bounded. If many local tokens exist, process a small batch and report partial failures.
  • Merge users transactionally where possible. If merge could touch many memberships later, plan batching with Convex scheduler.
  • During merge collisions, avoid overwriting an existing membership's availability records by accident. The source user's same-meeting voting set wins for that collision, while non-colliding memberships move normally.
  • Use indexes for all new list queries, especially memberships.by_user and emailIdentities.by_user.
